Unraid Configuration

Configuration of the file servers is fairly complex but, any tech-savvy person or homelab enthusiast should be able to understand the information on the following pages.

Definitions

The following definitions are useful to understand how Unraid works and does not work. The basics of a computer (memory, processors, etc.) is not discussed here and it is assumed you understand the basic construction of computers.

Term Definition
Array The set of disks that comprise the general storage in Unraid. This includes the parity drive and multiple data drives. Unlike RAID arrays the disks in Unraid do not need to be the same size.
Parity (parity drive) Parity is the data that is stored on the parity drive(s) that hold the information needed to serve or rebuild data from data drives in the array should one fail. The parity drive is the disk(s) designated as parity in the array.
Data drive Data drives in the array is where the files saved to the server are stored. Data drives must be the same size or smaller than the disk(s) designated as parity drive(s) in the array.
Shares Shares are folders that have been created and shared on the network. In Linux only the root-level folder can be shared (unlike Microsoft Windows servers). Shares are where user permissions are applied.
Plugins A main feature of Unraid is the plugin ecosystem that extends the functionality of Unraid. Plugins are installed using the "Apps" tab.
Container Containers are similar to "applications" that can be installed within Unraid to provide a service not related to Unraid. An example would be a webserver for a personal site (like this one). Currently, in Unraid only Docker containers are supported by default. However, LXC/LXD containers can be supported with the use of a plugin.
Virtual machine Unraid can natively support virtual machines. Similar to containers VMs can host services unrelated to Unraid but, managed from within Unraid.
